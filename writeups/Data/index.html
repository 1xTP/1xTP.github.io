1. Overview

Data is an "Easy" rated Linux machine from HackTheBox that serves as a great reminder of how a single unpatched web utility can lead to a full infrastructure compromise. In this challenge, I focus on exploiting a known directory traversal vulnerability in Grafana to perform post-exploitation reconnaissance on the underlying filesystem.

The path to root involves transitioning from a file read vulnerability to credential harvesting, and finally leveraging Docker group permissions to escape a containerized environment. My goal for this writeup is to document the transition from web-level access to host-level control.

2. Recon

Starting this machine, I ran a simple Nmap scan to see what ports were open. I like to use --min-rate 5000 to make things go quicker.

Port 3000 looked immediately interesting. While port 80 was just a standard landing page, port 3000 is the default home for Grafana. It's a common target, and I had a feeling the entry point would be there.

3. Exploiting Grafana LFI

After poking around the Grafana instance, I found it was running version 8.0.0, a version vulnerable to a Local File Inclusion (LFI). This is a classic example of why keeping internal tools patched is just as important as your public-facing web apps.

The Vulnerability

The specific flaw (CVE-2021-43798) allows an unauthenticated user to read arbitrary files by exploiting how Grafana handles plugin assets. I used a simple curl command to confirm I could reach the /etc/passwd file.

Hunting for the Database

Since I didn't have a password yet, I needed to find where Grafana stores its session and user data. By default, Grafana uses an SQLite database located at /var/lib/grafana/grafana.db. I used the LFI to download this database to my local machine for analysis.

After opening the database with sqlite3, I queried the user table. I found a hashed password for the user boris. After a quick conversion using grafana2hashcat I could crack the hash with Hashcat and gain valid credentials for SSH access.

4. Escaping the Container

Logging in as boris, the next step was to find a path to root. I usually start by checking my current user's sudo access.

The Discovery

Running the command sudo -l shows the Boris is allowed to run /usr/bin/docker exec * basically giving me root privileges in docker.

The Attack

I noticed a container already running on the system. Instead of just creating a new one, I decided to execute a shell inside the existing privileged container. By mounting the host's entire root filesystem into a directory inside the container, I could bypass the OS's file permissions entirely.

5. Conclusion

The "Data" machine highlights a critical lesson in modern infrastructure security: Internal tools are external targets. While the main web server on port 80 appeared secure, an auxiliary service (Grafana) provided the entry point.

This CTF demonstrated the "ripple effect" of a simple LFI. What started as a simple LFI allowed me to download a database, crack a user's password, and ultimately exploit Docker's operational design to seize full control of the host. For anyone securing similar environments, the takeaway is clear: enforce the Principle of Least Privilege for service accounts and keep your container daemon access strictly limited.
