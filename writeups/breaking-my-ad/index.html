1. Overview

In part 1, I built the "Gold Image". This image is a fully default AD setup. I made no effort to actually patch anything or make anything vulnerable.

The goal for this phase was to add vulnerabilities that I have seen in CTFs and real-world engagement reports often. I didn't want to just run the exploits, instead configure the vulnerability itself to understand exactly why they happen. I've added five attack points, ranging from default settings to complex misconfigurations.

2. The "I Didn't Even Do Anything" Problems

The scariest thing about these first two is that I didn't actually have to "break" anything. I just left the default Windows settings alone.

2.1 LLMNR & NBT-NS Poisoning

Windows is way too eager to find things on the network. If DNS fails, it falls back to legacy protocols like LLMNR to broadcast a "Who is this?" message to everyone.

The Config

By default the option to "Turn off multicast name resolution" is set to "not configured"

The Attack

I'll listen for these broadcasts on the Kali box and spoof a response to get the NTLMv2 hashes.

2.2 IPv6 DNS Takeover

Even if you aren't using IPv6, Windows favors it over IPv4. This is a massive blind spot.

The Config

The "misconfiguration" is just the default state of the network adapter.

The Attack

I’ll use mitm6 to pretend to be the DHCPv6 server and trick the Windows 11 client into sending authentication traffic to me.

3. Roasting

After the defaults, I moved on to identity-based attacks. These required me to manually weaken the user accounts I created with my PowerShell script.

3.1 AS-REP Roasting

This is a "lazy admin" move. I took one of my user accounts and manually disabled Kerberos pre-authentication.

The Config

I took one of my user accounts downgraded their security.

The Attack

Now, anyone can request a ticket that contains that user's password hash without needing a password first.

3.2 Kerberoasting

To practice kerberoasting, I needed a Service Principle Name (SPN) attached to a user account.

The Config

I created a dedicated service account (ssql) and used powershell to register an SPN for it.

The Attack

Now any valid user in the domain can request a service ticket for it, which I can then try to crack offline.

4. AD Certificate Services

This was the most complex setup. Active Directory Certificate Services (AD CS) is a beast, and misconfiguring it leads to total domain compromise (ESC1).

AD CS Misconfiguration (ESC1)

I installed the AD CS role on the Domain Controller. The vulnerability here isn't the CA itself, but a specific "Certificate Template" I created.

The Config

I created "Workstation" template that allows users to define their own identity in the certificate.

The Attack

Now that the certificate was active, I used Certipy-ad to scan the network and confirm the vulnerability was discoverable.

5. Conclusion

My lab is no longer a pristine environment and apparently never was in the first place as a default setup. Instead it’s a minefield.

I’ve verified that all 5 vulnerabilities are active and reachable from the Kali machine on the network. The infrastructure is done, the traps are set, and the users are weak.

It was very surprising to see the simple cause of some of the vulnerabilities I've exploited in the past during HackTheBox machines. Now that I know how these vulnerabilites are caused, I know how to keep them away.
